	<header class="entry-header">
		<h2 class="entry-title"><a href="https://marcobonzanini.com/2016/08/02/mastering-social-media-mining-with-python/" rel="bookmark">Mastering Social Media Mining with&nbsp;Python</a></h2>	</header><!-- .entry-header -->

	<div class="entry-content">
		<p><img data-attachment-id="237" data-permalink="https://marcobonzanini.com/2016/08/02/mastering-social-media-mining-with-python/book-cover/" data-orig-file="https://marcobonzanini.files.wordpress.com/2016/08/book-cover.jpg" data-orig-size="2077,2560" data-comments-opened="1" data-image-meta="{&quot;aperture&quot;:&quot;0&quot;,&quot;credit&quot;:&quot;&quot;,&quot;camera&quot;:&quot;&quot;,&quot;caption&quot;:&quot;&quot;,&quot;created_timestamp&quot;:&quot;0&quot;,&quot;copyright&quot;:&quot;&quot;,&quot;focal_length&quot;:&quot;0&quot;,&quot;iso&quot;:&quot;0&quot;,&quot;shutter_speed&quot;:&quot;0&quot;,&quot;title&quot;:&quot;&quot;,&quot;orientation&quot;:&quot;0&quot;}" data-image-title="book-cover" data-image-description="" data-image-caption="" data-medium-file="https://marcobonzanini.files.wordpress.com/2016/08/book-cover.jpg?w=243" data-large-file="https://marcobonzanini.files.wordpress.com/2016/08/book-cover.jpg?w=660" class="  wp-image-237 aligncenter" src="https://marcobonzanini.files.wordpress.com/2016/08/book-cover.jpg?w=345&#038;h=425" alt="book-cover" width="345" height="425" srcset="https://marcobonzanini.files.wordpress.com/2016/08/book-cover.jpg?w=345&amp;h=425 345w, https://marcobonzanini.files.wordpress.com/2016/08/book-cover.jpg?w=690&amp;h=850 690w, https://marcobonzanini.files.wordpress.com/2016/08/book-cover.jpg?w=122&amp;h=150 122w, https://marcobonzanini.files.wordpress.com/2016/08/book-cover.jpg?w=243&amp;h=300 243w" sizes="(max-width: 345px) 100vw, 345px"></p>
<p>Great news, my book on data mining for social media is finally out!</p>
<p>The title is <strong>Mastering Social Media Mining with Python</strong>. I&#8217;ve been working with Packt Publishing over the past few months, and in July the book has been finalised and released.</p>
<p>Links:</p>
<ul>
<li>ebook and paperback on <a href="https://www.packtpub.com/big-data-and-business-intelligence/mastering-social-media-mining-python">Packt Publishing</a> (the publisher)</li>
<li>ebook and paperback on <a href="https://www.amazon.com/Mastering-Social-Media-Mining-Python/dp/1783552018">Amazon.com</a> and <a href="https://www.amazon.co.uk/Mastering-Social-Media-Mining-Python/dp/1783552018">Amazon UK</a></li>
<li><a href="https://github.com/bonzanini/Book-SocialMediaMiningPython">Companion code for the book</a> on my GitHub</li>
<li><a href="https://goodreads.com/marcobonzanini">Author&#8217;s profile on goodreads</a> (inc. book ratings/reviews)</li>
</ul>
<p>As part of Packt&#8217;s <em>Mastering</em> series, the book assumes the readers already have some basic understanding of Python (e.g. <tt>for</tt> loops and classes), but more advanced concepts are discussed with examples. No particular experience with Social Media APIs and Data Mining is required. With 300+ pages, by the end of the book, the readers should be able to build their own data mining projects using data from social media and Python tools.</p>
<p>A bird&#8217;s eye view on the content:</p>
<ol>
<li>Social Media, Social Data and Python
<ul>
<li>Introduction on Social Media and Social Data: challenges and opportunities</li>
<li>Introduction on Python tools for Data Science</li>
<li>Overview on the use of public APIs to interact with social media platforms</li>
</ul>
</li>
<li>#MiningTwitter: Hashtags, Topics and Time Series
<ul>
<li>Interacting with the Twitter API in Python</li>
<li>Twitter data: the anatomy of a tweet</li>
<li>Entity analysis, text analysis, time series analysis on tweets</li>
</ul>
</li>
<li>Users, Followers, and Communities on Twitter
<ul>
<li>Analysing who follows whom</li>
<li>Mining your followers</li>
<li>Mining communities</li>
<li>Visualising tweets on a map</li>
</ul>
</li>
<li>Posts, Pages and User Interactions on Facebook
<ul>
<li>Interacting the Facebook Graph API in Python</li>
<li>Mining you posts</li>
<li>Mining Facebook Pages</li>
</ul>
</li>
<li>Topic analysis on Google Plus
<ul>
<li>Interacting with the Google Plus API in Python</li>
<li>Finding people and pages on G+</li>
<li>Analysis of notes and activities on G+</li>
</ul>
</li>
<li>Questions and Answers on Stack Exchange
<ul>
<li>Interacting with the StackOverflow API in Python</li>
<li>Text classification for question tags</li>
</ul>
</li>
<li>Blogs, RSS, Wikipedia, and Natural Language Processing
<ul>
<li>Blogs and web pages as social data Web scraping with Python</li>
<li>Basics of text analytics on blog posts</li>
<li>Information extraction from text</li>
</ul>
</li>
<li>Mining All the Data!
<ul>
<li>Interacting with many other APIs and types of objects</li>
<li>Examples of interaction with YouTube, Yelp and GitHub</li>
</ul>
</li>
<li>Linked Data and the Semantic Web
<ul>
<li>The Web as Social Media</li>
<li>Mining relations from DBpedia</li>
<li>Mining geo coordinates</li>
</ul>
</li>
</ol>
<p>The detailed table of contents is shown on the <a href="https://www.packtpub.com/big-data-and-business-intelligence/mastering-social-media-mining-python">Packt Pub&#8217;s page</a>. Chapter 2 is also offered as free sample.</p>
<p>Please have a look at the <a href="https://github.com/bonzanini/Book-SocialMediaMiningPython">companion code for the book</a> on my GitHub, so you can have an idea of the applications discussed in the book.</p>

	<header class="entry-header">
		<h2 class="entry-title"><a href="https://marcobonzanini.com/2015/11/21/adding-slack-notifications-to-a-luigi-pipeline-in-python/" rel="bookmark">Adding Slack Notifications to a Luigi Pipeline in&nbsp;Python</a></h2>	</header><!-- .entry-header -->

	<div class="entry-content">
		<p>In a previous article, I&#8217;ve described <a href="https://marcobonzanini.com/2015/10/24/building-data-pipelines-with-python-and-luigi/">how to build a data pipeline in Python</a> using <a href="https://github.com/spotify/luigi">Luigi</a>, a workflow manager written in Python and open sourced by Spotify. I also had the opportunity to give a short talk about Luigi at the local <a href="http://www.meetup.com/PyData-London-Meetup/">PyData London meetup</a> (see <a href="https://speakerdeck.com/marcobonzanini/let-luigi-do-the-plumbing-for-you-at-pydata-london-meetup">slides</a>).</p>
<p>One of the nice features of Luigi is the possibility of receiving e-mail notifications on error. While this is a useful feature, it&#8217;s tailored to errors only, so effectively you don&#8217;t know if the Luigi pipeline has completed its execution successfully, unless you manually check. As I wanted to extend the possibility of receiving a notification on Slack, also in case of success, I started looking around for the options. </p>
<p>I ended up developing my own solution: <a href="https://github.com/bonzanini/luigi-slack">https://github.com/bonzanini/luigi-slack</a>. This blog post is a brief overview on how to use this Python package with your Luigi pipeline.</p>
<h2>Getting started with luigi-slack</h2>
<p>From your organisation&#8217;s Slack page (e.g. yourname.slack.com) you can add a Bot integration. The setup is very quick, and you&#8217;ll receive a token that you&#8217;ll need to use to interact with the Slack API.</p>
<p>You can get the bleeding edge version of <tt>luigi-slack</tt>from the GitHub link above, but beware that this is a work in progress. A somewhat stable version is available from the cheese shop:</p>
<pre>pip install luigi-slack</pre>
<p>The key points of this package are:</p>
<ul>
<li>Support for Python 3</li>
<li>Easy-to-use interface</li>
</ul>
<p>Regarding the first point, the discussion on choosing Python 2 vs Python 3 is still never-ending and I&#8217;m not going there in this post. For a greenfield project, I prefer to use Python 3 rather than a version with a sunset date already decided. The support for Python 2 in luigi-slack is best-effort (and of course pull requests are always welcome).</p>
<p>In terms of easy-to-use interface, I borrowed the nice idea of using a context manager from <a href="https://github.com/hudl/luigi-monitor">luigi-monitor</a>, because it makes it easy to integrate the library with an existing pipeline.</p>
<p>For example, given the basic code to run a Luigi pipeline which ends with the task <tt>YourTaskClass</tt>:</p>
<pre class="brush: python; title: ; notranslate" title="">
import luigi

if __name__ == '__main__':
    luigi.run(main_task_cls=YourTaskClass)
</pre>
<p>All we need in order to have Slack notifications is to refactor as follows:</p>
<pre class="brush: python; title: ; notranslate" title="">
import luigi
from luigi_slack import SlackBot, notify

if __name__ == '__main__':
    slacker = SlackBot(token='my-token',
                       channels=['mychannel', 'anotherchannel'])
    with notify(slacker):
        luigi.run(main_task_cls=YourTaskClass)
</pre>
<h2>Configuration Options for luigi-slack</h2>
<p>The <tt>SlackBot</tt> takes a number of arguments. Besides the token, which allows you to connect to your organisation&#8217;s Slack, all the other parameters are optional:</p>
<ul>
<li><tt>channels</tt> (default empty list) is the list of channel names that you want to push the notifications to. For the channel name, you don&#8217;t need the initial <tt>#</tt> symbol. You can also deliver the notifications to a single account, by using the <tt>@username</tt> syntax</li>
<li><tt>events</tt> (default to [FAILURE]) is the list of event types, as defined in <tt>luigi_slack</tt>, that you want to track</li>
<li><tt>max_events</tt> (default to 5) is the max number of events of a given type that you want to report. With more than <tt>max_events</tt> events of the same type, a &#8220;please check logs&#8221; message is reported instead</li>
<li><tt>username</tt> (default to &#8220;Luigi-slack Bot&#8221;) is the screen name of your bot</li>
<li><tt>task_representation</tt> (default to <tt>str</tt>) is the function used to represent the task in the notification (see explanation below)</li>
</ul>
<p>In Luigi, representing a task as a string will print the <tt>task_id</tt> attribute of a <tt>luigi.Task</tt>, which include the class name as well as all the parameters. In other words, it looks like:</p>
<p>MyTask(param1=&#8221;some_value&#8221;, param2=&#8221;other_value&#8221;, your_secret_param=&#8221;your_secret_value&#8221;, &#8230;)</p>
<p>With a huge number of parameters that make the notification less readable, or with sensible parameters that you don&#8217;t want to send around in the Slack chat room, it makes sense to display the task a more conservative way. An example of custom string representation could be:</p>
<pre class="brush: python; title: ; notranslate" title="">
def custom_task_representation(task):
    return &quot;{}(...)&quot;.format(task.__class__.__name__)
</pre>
<p>Once we pass the function as <tt>task_representation</tt> argument of the SlackBot, the task will appear in the notifications as:</p>
<p>MyTask(&#8230;)</p>
<p>Keep in mind that an instance of a Luigi task is identified by the class name AND the value of its parameters, which is why the <tt>task_id</tt> include them all. In other words, with a more compact representation like the one proposed in the above snippet, you won&#8217;t be able to distinguish between tasks with the same class name, but different param values. You&#8217;ll need to customise the function based on your needs.</p>
<h2>Summary</h2>
<p>I&#8217;m developing a Python package to include Slack notification support to a Luigi pipeline, with a simple interface, a few optional configuration parameters, and minimal requirements in terms of refactoring.</p>
<p>The code is available at <a href="https://github.com/bonzanini/luigi-slack">https://github.com/bonzanini/luigi-slack</a>, and you can install the Python package with:</p>
<pre>pip install luigi_slack</pre>
<p>As this is a work in progress, it&#8217;s not widely tested, and the interface could change. Comments and pull requests are welcome.</p>

<article id="post-196" class="post-196 post type-post status-publish format-standard hentry category-big-data category-data-mining category-engineering category-python tag-big-data-2 tag-data-science tag-engineering-2 tag-luigi tag-python-2">
	
	<header class="entry-header">
		<h2 class="entry-title"><a href="https://marcobonzanini.com/2015/10/24/building-data-pipelines-with-python-and-luigi/" rel="bookmark">Building Data Pipelines with Python and&nbsp;Luigi</a></h2>	</header><!-- .entry-header -->

	<div class="entry-content">
		<p>As a data scientist, the emphasis of the day-to-day job is often more on the R&amp;D side rather than engineering. In the process of going from prototypes to production though, some of the early quick-and-dirty decisions turn out to be sub-optimal and require a decent amount of effort to be re-engineered. This usually slows down innovation, and generally speaking your project as a whole.</p>
<p>This post will discuss some experience in building data pipelines, e.g. extraction, cleaning, integration, pre-processing of data, in general all the steps that are necessary to prepare your data for your data-driven product. In particular, the focus in on <em>data plumbing</em>, and how a workflow manager like Luigi can come to the rescue, without getting in your way. With a minimal effort, the transition from prototype to production can be smoother.</p>
<p>You can find the code for the examples as <a href="https://gist.github.com/bonzanini/40774bf9348d35d9ea4f">GitHub Gist</a>.</p>
<h2>Early Days of a Prototype</h2>
<p>In the early days of a prototype, the data pipeline often looks like this:</p>
<pre>$ python get_some_data.py
$ python clean_some_data.py
$ python join_other_data.py
$ python do_stuff_with_data.py</pre>
<p>This is quite common when the data project is in its exploratory stage: you know that you&#8217;ll need some pre-processing, you think it&#8217;s going to be a quick hack, so you don&#8217;t bother with some engineering best practices, then the number of scripts grows and your data pipeline will come back and bite you.</p>
<p>This approach has the only advantage of being quick and hacky. On the downside, it&#8217;s tedious: every time you want to re-run the pipeline, you need to manually call the bunch of scripts in sequence. Moreover, if you&#8217;re sharing this prototype with a colleague, there is even more room for misinterpretation (&#8220;why can&#8217;t I do stuff with data?&#8221;&#8230; &#8220;did you clean it first?&#8221;, etc.)</p>
<p>The obvious hacky solution seems to be: let&#8217;s put everything in one script. After some quick refactoring, the <tt>do_everything.py</tt> script can look like this:</p>
<pre class="brush: python; title: ; notranslate" title="">
if __name__ == '__main__':
    get_some_data()
    clean_some_data()
    join_other_data()
    do_stuff_with_data()
</pre>
<p>This is fairly simple to run:</p>
<pre>$ python do_everything.py</pre>
<p>(Note: you could also put everything in a bash script, which calls the indiviaul bunch of scripts in sequence, but the shortcomings will be more or less the same)</p>
<h2>Boilerplate Code</h2>
<p>When moving towards a production-ready pipeline, there are a few more aspects to consider besides the run-everything code. In particular, error handling should be taken into account:</p>
<pre class="brush: python; title: ; notranslate" title=""> 
try:
    get_some_data()
except GetSomeDataError as e:
    # handle this
</pre>
<p>But if we chain all the individual tasks together, we end up with a Christmas tree of try/except:</p>
<pre class="brush: python; title: ; notranslate" title="">
try:
    get_some_data()
    try:
        clean_some_data()
        try:
            # you see where this is going...
        except EvenMoreErrors:
            # ...
    except CleanSomeDataError as e:
        # handle CleanSomeDataError
except GetSomeDataError as e:
    # handle GetSomeDataError
</pre>
<p>Another important aspect to consider is how to resume a pipeline. For example, if the first few tasks are completed, but then an error occurs half-way through, how do we re-run the pipeline without re-executing the initial successful steps?</p>
<pre class="brush: python; title: ; notranslate" title="">
# check if the task was already successful
if not i_got_the_data_already():
    # if not, run it
    try:
        get_some_date()
    except GetSomeDataError as e:
        # handle the error
</pre>
<h2>Moving to Luigi</h2>
<p><a href="https://github.com/spotify/luigi">Luigi</a> is a Python tool for workflow management. It has been developed at Spotify, to help building complex data pipelines of batch jobs. To install Luigi:</p>
<pre>$ pip install luigi</pre>
<p>Some of the useful features of Luigi include:</p>
<ul>
<li>Dependency management</li>
<li>Checkpoints / Failure recovery</li>
<li>CLI integration / parameterisation</li>
<li>Dependency Graph visualisation</li>
</ul>
<p>There are two core concepts to understand how we can apply Luigi to our own data pipeline: Tasks and Targets. A task is a unit of work, designed by extending the class <tt>luigi.Task</tt> and overriding some basic methods. The output of a task is a target, which can be a file on the local filesystem, a file on Amazon&#8217;s S3, some piece of data in a database etc.</p>
<p>Dependencies are defined in terms of inputs and outputs, i.e. if TaskB depends on TaskA, it means that the output of TaskA will be the input of TaskB.</p>
<p>Let&#8217;s look at a couple of template tasks:</p>
<pre class="brush: python; title: ; notranslate" title="">
# Filename: run_luigi.py
import luigi

class PrintNumbers(luigi.Task):

    def requires(self):
        return []

    def output(self):
        return luigi.LocalTarget(&quot;numbers_up_to_10.txt&quot;)

    def run(self):
        with self.output().open('w') as f:
            for i in range(1, 11):
                f.write(&quot;{}\n&quot;.format(i))

class SquaredNumbers(luigi.Task):

    def requires(self):
        return [PrintNumbers()]

    def output(self):
        return luigi.LocalTarget(&quot;squares.txt&quot;)

    def run(self):
        with self.input()[0].open() as fin, self.output().open('w') as fout:
            for line in fin:
                n = int(line.strip())
                out = n * n
                fout.write(&quot;{}:{}\n&quot;.format(n, out))
                
if __name__ == '__main__':
    luigi.run()

</pre>
<p>This code showcases two tasks: <tt>PrintNumbers</tt>, that writes the number from 1 to 10 into a file called <tt>numbers_up_to_10.txt</tt>, one number per line, and <tt>SquaredNumbers</tt>, that reads the such file and outputs a list of pairs number-square into <tt>squares.txt</tt>, also one pair per line.</p>
<p>To run the tasks:</p>
<pre>$ python run_luigi.py SquaredNumbers --local-scheduler</pre>
<p>Luigi will take care of checking the dependencies between tasks, see that the input of <tt>SquaredNumbers</tt> is not there, so it will run the <tt>PrintNumbers</tt> task first, then carry on with the execution.</p>
<p>The first argument we&#8217;re passing to Luigi is the name of the last task in the pipeline we want to run. The second argument simply tells Luigi to use a local scheduler (more on this later).</p>
<p>You could also use the <tt>luigi</tt> command:</p>
<pre>$ luigi -m run_luigi.py SquaredNumbers --local-scheduler</pre>
<h2>Anatomy of a Task</h2>
<p>To create a Luigi task we simply need to create a class whose parent is <tt>luigi.Task</tt>, and override some methods. In particular:</p>
<ul>
<li><tt>requires()</tt> should return the list of dependencies for the given task &#8212; in other words a list of tasks</li>
<li><tt>output()</tt> should return the target for the task (e.g. a LocalTarget, a S3Target, etc.)</li>
<li><tt>run()</tt> should contain the logic to execute</li>
</ul>
<p>Luigi will check the return values of <tt>requires()</tt> and <tt>output()</tt> and build the dependency graph accordingly.</p>
<h2>Passing Parameters</h2>
<p>Hard-coding filenames and config values is generally speaking an anti-pattern. Once you&#8217;ve understood the structure and the dynamics of your task, you should look into parameterising all the configuration aspects so that you can dynamically call the same script with different arguments. </p>
<p>The class <tt>luigi.Parameter()</tt> is the place to look into. Each Luigi task can have a number of parameters. Let&#8217;s say for example that we want to modify the previous example to support a custom number. As the parameter we&#8217;re using with the <tt>range()</tt> function is an integer, we can use <tt>luigi.IntParameter</tt> rather than the default parameter class. This is how the modified tasks can look like:</p>
<pre class="brush: python; title: ; notranslate" title="">
class PrintNumbers(luigi.Task):
    n = luigi.IntParameter()

    def requires(self):
        return []

    def output(self):
        return luigi.LocalTarget(&quot;numbers_up_to_{}.txt&quot;.format(self.n))

    def run(self):
        with self.output().open('w') as f:
            for i in range(1, self.n+1):
                f.write(&quot;{}\n&quot;.format(i))

class SquaredNumbers(luigi.Task):
    n = luigi.IntParameter()

    def requires(self):
        return [PrintNumbers(n=self.n)]

    def output(self):
        return luigi.LocalTarget(&quot;squares_up_to_{}.txt&quot;.format(self.n))

    def run(self):
        with self.input()[0].open() as fin, self.output().open('w') as fout:
            for line in fin:
                n = int(line.strip())
                out = n * n
                fout.write(&quot;{}:{}\n&quot;.format(n, out))
</pre>
<p>To call the <tt>SquaredNumbers</tt> tasks up to, say, 20:</p>
<pre>$ python run_luigi.py SquaredNumbers --local-scheduler --n 20</pre>
<p>Parameters can also have default values, e.g.</p>
<pre class="brush: python; title: ; notranslate" title="">
n = luigi.IntParameter(default=10)
</pre>
<p>so in this way, if you don&#8217;t specify the <tt>--n</tt> argument, it will default to 10.</p>
<p><a href="https://gist.github.com/bonzanini/40774bf9348d35d9ea4f">Sample code as GitHub Gist</a></p>
<h2>Local vs Global Scheduler</h2>
<p>So far, we&#8217;ve used the <tt>--local-scheduler</tt> option to run Luigi tasks with a local scheduler. This is useful for development, but in a production environment we should make use of the centralised scheduler (see the docs on the <a href="http://luigi.readthedocs.org/en/latest/central_scheduler.html">scheduler</a>).</p>
<p>This has a few advantages: </p>
<ul>
<li>avoid running two instances of the same task simultaneously</li>
<li>nice web-based visualisation</li>
</ul>
<p>You can run the Luigi scheduler daemon in the foreground with:</p>
<pre>$ luigid</pre>
<p>or in the background with:</p>
<pre>$ luigid --background</pre>
<p>It will default to port 8082, so you can point your browser to <a href="http://localhost:8082">http://localhost:8082</a> to access the visualisation.</p>
<p>With the global Luigi scheduler running, we can re-run the code without the option for the local scheduler:</p>
<pre>$ python run_luigi.py SquaredNumbers --n [BIG_NUMBER]</pre>
<p>As the sample code will run in milliseconds, if you want to have a chance to switch to the browser and see the dependency graph while the tasks are still running, you should probably use a big number like 10,000,000 or more for the <tt>--n</tt> option.</p>
<p>This is a cropped screenshot of the dependency graph:</p>
<p><a href="https://marcobonzanini.files.wordpress.com/2015/10/dependency-graph-screenshot.png"><img data-attachment-id="197" data-permalink="https://marcobonzanini.com/2015/10/24/building-data-pipelines-with-python-and-luigi/dependency-graph-screenshot/" data-orig-file="https://marcobonzanini.files.wordpress.com/2015/10/dependency-graph-screenshot.png" data-orig-size="1472,420" data-comments-opened="1" data-image-meta="{&quot;aperture&quot;:&quot;0&quot;,&quot;credit&quot;:&quot;&quot;,&quot;camera&quot;:&quot;&quot;,&quot;caption&quot;:&quot;&quot;,&quot;created_timestamp&quot;:&quot;0&quot;,&quot;copyright&quot;:&quot;&quot;,&quot;focal_length&quot;:&quot;0&quot;,&quot;iso&quot;:&quot;0&quot;,&quot;shutter_speed&quot;:&quot;0&quot;,&quot;title&quot;:&quot;&quot;,&quot;orientation&quot;:&quot;0&quot;}" data-image-title="dependency-graph-screenshot" data-image-description="" data-image-caption="" data-medium-file="https://marcobonzanini.files.wordpress.com/2015/10/dependency-graph-screenshot.png?w=300" data-large-file="https://marcobonzanini.files.wordpress.com/2015/10/dependency-graph-screenshot.png?w=660" src="https://marcobonzanini.files.wordpress.com/2015/10/dependency-graph-screenshot.png?w=300&#038;h=86" alt="dependency-graph-screenshot" width="300" height="86" class="alignnone size-medium wp-image-197" srcset="https://marcobonzanini.files.wordpress.com/2015/10/dependency-graph-screenshot.png?w=300 300w, https://marcobonzanini.files.wordpress.com/2015/10/dependency-graph-screenshot.png?w=600 600w, https://marcobonzanini.files.wordpress.com/2015/10/dependency-graph-screenshot.png?w=150 150w" sizes="(max-width: 300px) 100vw, 300px" /></a></p>
<h2>Summary</h2>
<p>We have described the definition of data pipelines using Luigi, a workflow manager written in Python. Luigi provides a nice abstraction to define your data pipeline in terms of tasks and targets, and it will take care of the dependencies for you.</p>
<p>In terms of code re-use, and with the mindset of going from prototype to production, I&#8217;ve found very helpful to define the business logic of the tasks in separate <a href="https://marcobonzanini.com/2015/07/01/how-to-develop-and-distribute-python-packages/">Python packages</a> (i.e. with a setup.py file). In this way, from your Luigi script you can simply <tt>import your_package</tt> and call it from there.</p>
<p>A task can produce multiple files as output, but if that&#8217;s your case, you should probably verify if the task can be broken down into smaller units (i.e. multiple tasks). Do all these outputs logically belong together? Do you have dependencies between them? If you can&#8217;t break the task down, I&#8217;ve found it simpler/useful just to define the <tt>output()</tt> as a log file with the names and the timestamps of all the individual files created by the task itself. The log file name can be formatted as <tt>TaskName_timestamp_param1value_param2value_etc</tt>.</p>
<p>Using a workflow manager like Luigi is in general helpful because it handles dependencies, it reduces the amount of boilerplate code that is necessary for parameters and error checking, it manages failure recovery and overall it forces us to follow a clear pattern when developing the data pipeline.</p>
<p>It&#8217;s also important to consider its limitations:</p>
<ul>
<li>It was built for batch jobs, it&#8217;s probably not useful for near real-time processing</li>
<li>It doesn&#8217;t trigger the execution for you, you still need to run the data pipeline (e.g. via a cronjob)</li>
</ul>

	<header class="entry-header">
		<h2 class="entry-title"><a href="https://marcobonzanini.com/2015/07/09/getting-started-with-apache-spark-and-python-3/" rel="bookmark">Getting Started with Apache Spark and Python&nbsp;3</a></h2>	</header><!-- .entry-header -->

	<div class="entry-content">
		<p><a href="https://spark.apache.org/">Apache Spark</a> is a cluster computing framework, currently one of the most actively developed in the open-source Big Data arena. It aims at being a general engine for large-scale data processing, supporting a number of platforms for cluster management (e.g. YARN or Mesos as well as Spark native) and a variety of distributed storage systems (e.g. HDFS or Amazon S3). </p>
<p>More interestingly, at least from a developer&#8217;s perspective, it supports a number of programming languages. Since the latest version 1.4 (June 2015), Spark supports R and Python 3 (to complement the previously available support for Java, Scala and Python 2).</p>
<p>This article is a brief introduction on how to use Spark on Python 3.</p>
<h2>Quick Start</h2>
<p>After downloading a binary version of Spark 1.4, we can extract it in a custom folder, e.g. <tt>~/apps/spark</tt>, which we&#8217;ll call $SPARK_HOME:</p>
<pre>export SPARK_HOME=~/apps/spark</pre>
<p>This folder contains several Spark commands (in <tt>$SPARK_HOME/bin</tt>) as well as examples of code (in <tt>$SPARK_HOME/examples/src/main/YOUR-LANGUAGE</tt>).</p>
<p>We can run Spark with Python in two ways: using the interactive shell, or submitting a standalone application.</p>
<p>Let&#8217;s start with the interactive shell, by running this command:</p>
<pre>$SPARK_HOME/bin/pyspark</pre>
<p>You will get several messages on the screen while the shell is loading, and at the end you should see the Spark banner:</p>
<pre>Welcome to
      ____              __
     / __/__  ___ _____/ /__
    _\ \/ _ \/ _ `/ __/  '_/
   /__ / .__/\_,_/_/ /_/\_\   version 1.4.0
      /_/

Using Python version 2.7.5 (default, Mar  9 2014 22:15:05)
SparkContext available as sc, HiveContext available as sqlContext.
&gt;&gt;&gt;</pre>
<p>The <tt>&gt;&gt;&gt;</tt> prompt is the usual Python prompt, as effectively we are using a Python interactive shell.</p>
<p><tt>SparkContext</tt> and <tt>HiveContext</tt> are Spark concepts that we&#8217;ll briefly explain below. The interactive shell is telling us that these two contexts have been initialised and are available as <tt>sc</tt> and <tt>sqlContext</tt> in this session. The shell is also telling us that we&#8217;re using Python 2.7!?</p>
<p><em>But I want to use Python 3!</em></p>
<p>Long story short, Python 2 is still the default option in Spark, which you can see if you open the <tt>pyspark</tt> script with an editor (it&#8217;s a shell script). You can simply override this behaviour by setting an environment variable:</p>
<pre>export PYSPARK_PYTHON=python3</pre>
<p>Once you re-run the interactive shell, the Spark banner should be updated to reflect your version of Python 3. </p>
<h2>Some Basic Spark Concepts</h2>
<p>Two Spark concepts have already been mentioned above:</p>
<ul>
<li><tt>SparkContext</tt>: it&#8217;s an object that represents a connection to a computing cluster &#8211; an application will access Spark through this context</li>
<li><tt>HiveContext</tt>: it&#8217;s an instance of the Spark SQL engine, that integrates data stored in Hive (not used in this article)</li>
</ul>
<p>Another core concept in Spark is the Resilient Distributed Dataset (RDD), an immutable distributed collection of objects. Each RDD is split into partitions, which might be processed on different nodes of a cluster.</p>
<p>RDDs can be loaded from external sources, e.g. from text files, or can be transformed into new RDDs. </p>
<p>There are two types of operation that can be performed over a RDD:</p>
<ul>
<li>a <em>transformation</em> will leave the original RDD intact and create a new one (RDD are immutable); an example of transformation is the use of a filter</li>
<li>an <em>action</em> will compute a result based on the RDD, e.g. counting the number of lines in a RDD
</ul>
<h2>Running an Example Application</h2>
<p>Running the interactive shell can be useful for interactive analysis, but sometimes you need to launch a batch job, i.e. you need to submit a stand-alone application.</p>
<p>Consider the following code and save it as <tt>line_count.py</tt>:</p>
<pre class="brush: python; title: ; notranslate" title="">
from pyspark import SparkContext

import sys

if __name__ == '__main__':

    fname = sys.argv[1]
    search1 = sys.argv[2].lower()
    search2 = sys.argv[3].lower()

    sc = SparkContext(&quot;local&quot;, appName=&quot;Line Count&quot;)
    data = sc.textFile(fname)

    # Transformations
    filtered_data1 = data.filter(lambda s: search1 in s.lower())
    filtered_data2 = data.filter(lambda s: search2 in s.lower())
    # Actions
    num1 = filtered_data1.count()
    num2 = filtered_data2.count()

    print('Lines with &quot;%s&quot;: %i, lines with &quot;%s&quot;: %i' % (search1, num1, search2, num2))
</pre>
<p>The application will take three parameters from the command line (via <tt>sys.argv</tt>), namely a file name and two search terms.</p>
<p>The <tt>sc</tt> variable contains the SparkContext, initialised as local context (we&#8217;re not using a cluster).</p>
<p>The <tt>data</tt> variable is the RDD, loaded from an external resource (the aforementioned file).</p>
<p>What follows the data import is a series of transformations and actions. The basic idea is to simply count how many lines contain the given search terms.</p>
<p>For this example, I&#8217;m using <a href="https://marcobonzanini.com/2015/03/23/mining-twitter-data-with-python-part-4-rugby-and-term-co-occurrences/">a data-set of tweets</a> downloaded for a previous article, stored in <tt>data.json</tt> one tweet per line.</p>
<p>I can now launch (submit) the application with:</p>
<pre>$SPARK_HOME/bin/spark-submit line_count.py data.json \#ita \#eng</pre>
<p>Within a lot of debugging information, the application will print out the final count:</p>
<pre>Lines with "#ita": 1339, lines with "#eng": 2278</pre>
<p>Notice the use of the backslash from the command-line, because we need to escape the <tt>#</tt> symbol: effectively the search terms are <tt>#ita</tt> and <tt>#eng</tt>.</p>
<p>Also notice that we don&#8217;t have information about repeated occurrences of the search terms, nor about partial matches (e.g. &#8220;#eng&#8221; will also match &#8220;#england&#8221;, etc.): this example just showcases the use of transformations and actions.</p>
<h2>Summary</h2>
<p>Spark now supports Python 3 :)</p>
